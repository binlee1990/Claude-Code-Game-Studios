# game-concept.md — 修仙放置挂机刷宝 RPG

> 版本：v2.0 Ultimate Concept  
> 目标：在“修仙放置挂机 RPG”基础上，扩展为一个功能完备、玩法复杂、长期可扩展的中文半放置刷宝队伍 RPG。  
> 主开发路线：**Godot 4 自研主工程 + 数据驱动 + 单机优先 + 可扩展内容库**。

---

## 1. 一句话概念

一款中文原创修仙题材的 **半放置刷宝队伍 RPG**：玩家从凡人闭关开始，通过自动修炼、自动战斗、装备刷宝、队伍 Build、生活技能、城镇宗门经营、秘境远征、天劫突破、飞升轮回和法则重置，逐步成长为统御诸天的修行者。

核心体验不是“单纯看数字增长”，而是：

```text
挂机产出资源
→ 自动战斗刷装备
→ 调整角色、职业、技能、战法、阵法、宠物、装备 Build
→ 推进更高地图和副本
→ 城镇宗门生产反哺战斗
→ 境界突破 / 飞升 / 轮回 / 合道
→ 解锁新系统、新世界层、新自动化能力
```

## 3. 玩家幻想

玩家扮演一名从凡人村落走出的修行者。最初只能闭关修炼、积攒灵气；随后解锁洞府、灵田、丹炉、战斗队伍、宗门、秘境、灵宠、法宝、拍卖、天劫、飞升和诸天远征。

理想体验是：

> “我离开时，世界仍在运转；我回来时，角色更强、仓库更多、装备更好、宗门生产完成，并且出现了新的选择。”

玩家长期追求包括：

- 更高境界；
- 更强队伍；
- 更优 Build；
- 更稀有装备；
- 更完整图鉴；
- 更高自动化；
- 更深世界层；
- 更复杂但可管理的宗门经济；
- 更极限的 Boss、兽潮、远征、深渊层数。

## 4. 核心设计支柱

### 4.1 数字增长就是快乐

灵气、修为、灵石、战力、掉落率、生产效率、离线收益持续增长；但增长要有阶段、瓶颈、突破和重置，避免单调。

### 4.2 放置不是无操作，而是低频高价值决策

玩家不需要逐秒操作。主要决策包括：

- 选择挂机区域；
- 配置队伍；
- 设置战法；
- 筛选装备；
- 规划建筑；
- 分配居民；
- 准备突破；
- 选择飞升保留项；
- 设计远征路线；
- 决定挑战哪些高风险内容。

### 4.3 刷宝提供惊喜

中后期核心驱动力是装备、词条、法宝、宠物、卷轴、稀有材料和特殊 Build，而不是只有资源数字上涨。

### 4.4 城镇宗门是后勤系统，不是装饰

宗门建筑、居民、生产链、科技树、任务堂、拍卖行必须反哺战斗、修炼、突破和远征。

### 4.5 多层重置制造长期目标

突破、飞升、轮回、合道是不同层级的重置：

- 突破：小重置，提升境界和资源上限；
- 飞升：中重置，进入新世界层；
- 轮回：大重置，保留命格和账号加成；
- 合道：终局重置，改变世界规则。

### 4.6 渐进叙事展开

开局只给“修炼”。系统逐步出现，世界逐步展开。玩家不是进入一个菜单地狱，而是发现一个越来越大的修仙世界。

### 4.7 子玩法服务主循环

兽潮、远征、阵盘、卡牌试炼、爬塔、Boss 副本都必须回流到主循环，产出装备、材料、法则点、宠物、卷轴或 Build 组件。

### 4.8 自动化是成长奖励

自动拾取、自动分解、自动炼丹、自动派遣、自动突破、自动轮回等能力应通过进度解锁，而不是一开始全开放。

### 4.9 单机经济优先

第一版不做多人市场、不做实时联机、不做交易经济。全部资源循环必须在单机内自洽。

### 4.10 数据驱动与可扩展

所有角色、装备、技能、词条、建筑、配方、事件、区域、Boss、法则都应数据驱动，便于 AI 辅助生成、长期扩展和 Mod。

## 7. 核心循环

### 7.1 10 秒循环

```text
资源跳动
→ 战斗日志刷新
→ 技能触发
→ 掉落提示
→ 玩家观察是否有异常或稀有掉落
```

### 7.2 30 秒循环

```text
灵气/修为自动增长
→ 点击或短时加速修炼
→ 查看当前产出倍率
→ 发现下一个可购买升级
```

### 7.3 5 分钟循环

```text
自动战斗若干轮
→ 获得经验、材料、装备
→ 等级提升或装备替换
→ 产出效率上升
```

### 7.4 30 分钟循环

```text
生产链完成一批资源
→ 药材转丹药 / 矿石转装备 / 材料转符箓
→ 建筑升级或新配方解锁
→ 下一阶段瓶颈转移
```

### 7.5 数小时循环

```text
准备突破材料
→ 挑战天劫或 Boss
→ 境界突破
→ 部分资源重置
→ 保留天赋、里程碑、自动化
→ 解锁新系统
```

### 7.6 多日循环

```text
完成一轮飞升 / 轮回
→ 进入新世界层
→ 旧内容高效自动化
→ 新资源、新敌人、新法则、新 Build 出现
```

### 7.7 赛季/章节循环

```text
开放新秘境、新职业、新法宝、新事件规则
→ 玩家构筑新 Build
→ 赛季结束后部分内容沉淀为常驻内容
```

## 11. MVP 定义（阶段 1：最小挂机闭环）

MVP 不追求“完整游戏”，只验证基础手感、离线收益、大数值、自动战斗和存档。

| 功能 | MVP 目标 |
|---|---|
| 灵气自动增长 | 每秒产出灵气 |
| 点击修炼 | 手动点击获得额外灵气 |
| 自动修炼 | 每秒自动获得修为或灵气 |
| 大数值显示 | 支持至少 1e30 级显示 |
| 存档系统 | 自动保存、读取、版本号 |
| 离线收益 | 根据时间戳补发灵气、经验、基础材料 |
| 简单自动战斗 | 自动刷怪，获得经验/材料 |
| 等级提升 | 等级越高，产出越快 |
| 简单区域 | 3 个挂机区域，敌人和收益不同 |
| 基础 UI | 修炼、战斗、资源、存档、离线结算页面 |

MVP 明确不包含：

- 抽卡；
- 多人；
- 联机交易；
- 宗门完整经营；
- 秘境远征；
- 复杂装备词条海；
- 宠物系统；
- 兽潮模式；
- 大型剧情；
- 付费系统。

## 13. 技术考量

### 13.1 离线收益

离线收益必须通过“退出时间戳 → 返回时间戳”的差值计算，不能依赖实时后台 `_process`。Web 版本尤其需要注意：浏览器标签页不活跃时项目可能暂停，Godot 的 `_process()` / `_physics_process()` 不会继续运行。

### 13.2 大数值

需要 Decimal/BigNumber 类型，支持：

- 加减乘除；
- 幂运算；
- 对数比较；
- 科学计数法；
- 中文缩写显示；
- 大数排序；
- 存档序列化。

### 13.3 数据驱动

所有内容必须配置化：

```text
characters
classes
skills
items
affixes
loot_tables
enemies
zones
buildings
recipes
pets
formations
prestige_layers
events
achievements
```

### 13.4 存档

存档必须包含：

- 当前资源；
- 解锁状态；
- 角色状态；
- 背包；
- 建筑；
- 队伍配置；
- 战法配置；
- 离线时间戳；
- 重置层状态；
- 内容包版本；
- 迁移版本号。

### 13.5 战斗模拟

战斗应支持两种模式：

| 模式 | 用途 |
|---|---|
| 可视化战斗 | 在线展示、Boss、重要挑战 |
| 批量模拟 | 离线收益、扫荡、长时间挂机结算 |

### 13.6 可维护性

复杂游戏最容易失败在“系统互相污染”。必须采用模块化边界：

```text
ResourceSystem 不直接知道 UI
CombatSystem 不直接修改存档
LootSystem 不直接控制战斗
SaveSystem 只保存状态，不计算收益
FormulaEngine 统一处理公式
EventBus 统一发事件
```

## 14. 设计边界与反膨胀规则

即使目标是“最完备、最复杂”，也不能无规则堆系统。每个系统必须满足至少一项：

| 合格条件 | 说明 |
|---|---|
| 提供更好装备 | 增强刷宝惊喜 |
| 提供更强 Build | 增强策略配置 |
| 提供更高自动化 | 减少重复劳动 |
| 提供更深成长 | 支撑长期目标 |
| 提供新资源瓶颈 | 改变经济循环 |
| 提供新世界感 | 强化修仙幻想 |
| 提供新挑战 | 检验已有系统 |

不满足这些条件的系统应进入“延后/删除”列表。

## 15. 当前最终定位

本项目不是简单的“修仙点击器”，而是：

> **以修仙世界观包装的半放置刷宝队伍 RPG。**  
> 它用放置保证轻负担，用刷宝保证惊喜，用 Build 保证策略，用宗门保证经济纵深，用飞升轮回保证长期目标，用秘境和兽潮保证短局变化，用渐进叙事保证世界感。

最终目标：

```text
让玩家每天回来都有三件事可做：
1. 看见挂机带来的成长；
2. 做出几个有价值的配置决策；
3. 解锁或接近一个新的系统、境界、装备、区域或故事。
```

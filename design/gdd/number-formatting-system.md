# 数值格式化系统 (Number Formatting System)

> **Status**: Designed
> **Author**: binlee1990 + agents
> **Last Updated**: 2026-05-03
> **Implements Pillar**: 4.1 数字增长就是快乐

## Summary

数值格式化系统将 BigNumber 的 `{mantissa, exponent}` 内部表示转为玩家可读的中文缩写或科学计数法字符串。它是所有 UI 数值显示的统一出口——没有它，玩家看到的是一串无法理解的数字。

> **Quick reference** — Layer: `Core Data` · Priority: `MVP` · Key deps: `大数值系统`

## Overview

数值格式化系统是游戏所有数值显示的统一出口。它将大数值系统的内部表示 `{mantissa, exponent}` 转换为玩家可读的显示字符串，支持三个格式层级：

1. **直接数字**（< 10^4）：显示完整数值，带千分位分隔，如 `999`、`9,999`
2. **中文单位**（10^4 ~ 10^48）：使用万、亿、兆、京、垓、秭、穰、沟、涧、正、载、极共 12 个单位，保留 3 位有效数字，如 `1.53万`、`5.67兆`
3. **科学计数法**（> 10^48）：超出中文单位范围后自动切换，如 `1.23e50`

格式化为**单向操作**（BigNumber → 字符串），不负责反向解析。MVP 阶段固定中文优先格式，不支持玩家切换。本系统被 HUD、资源面板、战斗日志等所有需要显示数值的 UI 组件调用。

## Player Fantasy

玩家不会直接"操作"数值格式化，但每次看到屏幕上的数字跳动时，他们感受到的是修仙世界观的沉浸：

- **早期**（百、千）：直观的数字增长，"我的灵气从 500 涨到了 3,000"，成就感来自数字本身。
- **中期突破感**（万、亿、兆）：当数字从 `9,999` 跳到 `1.00万`，从 `9999.99万` 跳到 `1.00亿`——每一次单位跃迁都是"破境"的视觉映射。中文单位让修仙世界观与数值体系融为一体。
- **后期超脱感**（京、垓、秭、科学计数法）：当数值进入冷僻的中文单位甚至科学计数法时，玩家感受到的是"凡人无法理解的境界"——数字越大越超然，强化"诸天至尊"的终极幻想。

核心体验：**数字永远在增长，单位永远在跃迁，没有天花板**。

### 格式选择规则

| 条件 | 格式 | 示例 |
|------|------|------|
| `exponent < 4` | 直接数字 + 千分位逗号 | `999`、`9,999` |
| `4 ≤ exponent < 52` | 中文单位格式 | `1.50万`、`567万`、`3.14亿`、`9990极` |
| `exponent ≥ 52` | 科学计数法 | `1.23e52` |
| `value == 0` | 固定显示 | `0` |
| `value == MAX` | 固定显示 | `MAX` |

### 精度规则

中文单位格式计算 `display_mantissa = BigNumber.mantissa × 10^(exponent - unit_threshold)`（结果 ∈ [1.0, 10000.0)），按以下规则取精度：

| display_mantissa 范围 | 小数位数 | 有效数字 | 示例 |
|---|---|---|---|
| [1.0, 10.0) | 2 位 | 3 | `1.50万` |
| [10.0, 100.0) | 1 位 | 3 | `15.3万` |
| [100.0, 10000.0) | 0 位 | 3-4 | `567万`、`5678万` |

科学计数法固定 2 位小数：`M.MMeN`（如 `1.23e50`）。

### 千分位分隔

直接数字格式中，每 3 位加逗号：`9,999`、`123,456`、`1,234`。

### API 表面

- `format(value: BigNumber) -> String` — 默认格式化（中文优先）
- `format_scientific(value: BigNumber) -> String` — 强制科学计数法
- `format_raw(value: BigNumber) -> String` — 直接数字（仅小数值有效，大值回退科学计数法）
- `get_display_unit(value: BigNumber) -> String` — 返回当前单位后缀（如 "万"、"亿"）

## Formulas

### F1: 单位阈值选择

```
unit_threshold(e) = floor(e / 4) × 4
```

| 变量 | 类型 | 范围 | 说明 |
|------|------|------|------|
| e | int | [0, 308] | BigNumber.exponent |
| unit_threshold | int | [0, 308] | 对应单位的 10 的幂次 |

选择规则：
- `unit_threshold < 4` → 直接数字格式
- `4 ≤ unit_threshold ≤ 48` → 中文单位，单位索引 = `(unit_threshold - 4) / 4`
- `unit_threshold > 48` → 科学计数法

### F2: 显示尾数计算

```
display_mantissa = m × 10^(e - T)
```

| 变量 | 类型 | 范围 | 说明 |
|------|------|------|------|
| m | float | [1.0, 10.0) | BigNumber.mantissa |
| e | int | [0, 308] | BigNumber.exponent |
| T | int | [4, 48] | 单位阈值 |
| display_mantissa | float | [1.0, 10000.0) | 单位前的显示数值 |

### F3: 小数位数决策

```
if display_mantissa < 10:    decimals = 2
elif display_mantissa < 100:  decimals = 1
else:                         decimals = 0
```

| 变量 | 类型 | 范围 | 说明 |
|------|------|------|------|
| decimals | int | {0, 1, 2} | 保留小数位数 |

### F4: 科学计数法格式 (e ≥ 52)

```
formatted = "{m:.2f}e{e}"
```

示例：`{1.234, 50}` → `"1.23e50"`（注意：e=50 < 52，实际用中文单位 "123极"）

### F5: 直接数字格式 (e < 4)

```
integer_value = round(m × 10^e)
formatted = add_thousand_separators(integer_value)
```

示例：`{1.234, 3}` → `round(1234)` → `"1,234"`

## Edge Cases

- **零值**：`BigNumber.ZERO {0.0, 0}` → 固定输出 `"0"`，不走任何格式化逻辑。
- **MAX 值**：`BigNumber.MAX {9.999, 308}` → 固定输出 `"MAX"`，表示已达数值上限。调用 `format_scientific()` 时也返回 `"MAX"`。
- **NaN/Inf 尾数**：若 BigNumber 的 mantissa 为 NaN 或 Inf（防御性检查），返回 `"0"`。正常使用中不应出现此情况（BigNumber 运算已钳位），但格式化器不应崩溃。
- **舍入后跨格式边界（直接数字 → 中文单位）**：`{9.9999, 3}` → round(9999.9) = 10000 → 切换到中文单位 → `"1.00万"`。规则：直接数字格式舍入后 ≥ 10000，重新以中文单位格式处理。
- **舍入后跨单位边界（中文单位 → 下一单位）**：`{9.9999, 7}` → T=4, display=9999.9 → 舍入=10000 → 单位提升至 T=8 → display=0.99999 → 钳位到 1.00 → `"1.00亿"`。规则：display_mantissa 舍入后 ≥ 10000，单位提升 4 级，重新计算。
- **舍入后跨入科学计数法（极 → 科学）**：`{9.9999, 51}` → T=48, display=9999.9 → 舍入=10000 → 新 T=52 > 48 → `"1.00e52"`。
- **exponent 恰好等于单位阈值**：`{1.0, 4}` → `"1.00万"` ✓；`{1.0, 8}` → `"1.00亿"` ✓。预期行为，无特殊处理。
- **display_mantissa 舍入到刚好 1.0**：值略低于某个单位（如 9,990万 ≈ 0.999亿）→ 显示为 `"1.00亿"`。规则：display_mantissa < 1.0 时（由单位提升导致），强制钳位到 1.0。
- **format() 被高频调用**：HUD 每帧可能调用多次。格式化器为无状态纯函数，无副作用，可安全高频调用。缓存由调用方（UI 层）负责，格式化器本身不做缓存。

## Dependencies

### 上游依赖

| 系统 | 方向 | 依赖性质 | 数据接口 |
|------|------|---------|---------|
| 大数值系统 | 上游硬依赖 | 必需 | 读取 `mantissa`、`exponent`、`is_zero()`、`is_max()` 进行格式化 |

### 下游消费者

| 系统 | 方向 | 依赖性质 | 数据接口 |
|------|------|---------|---------|
| HUD 系统 | 下游调用 | 必需 | 调用 `format()` 显示灵气、修为、战力等数值 |
| 资源系统 | 下游间接 | 通过 HUD | 资源数值经 HUD 调用 `format()` |
| 属性系统 | 下游间接 | 通过 HUD | 角色属性经 HUD 调用 `format()` |
| 离线收益结算系统 | 下游调用 | 必需 | 结算摘要中调用 `format()` 显示离线获得资源 |
| 战斗日志系统 | 下游调用 | 必需 | 伤害、治疗等数值经 `format()` 显示 |

双向一致性：大数值系统 GDD 的 Dependencies 节已列出"数值格式化系统"为下游消费方，与本节一致。HUD 系统 GDD 需在 Dependencies 节中列出"数值格式化系统"为上游依赖。

## Tuning Knobs

| 参数 | 当前值 | 安全范围 | 增大影响 | 减小影响 |
|------|--------|---------|---------|---------|
| `SCIENTIFIC_THRESHOLD` | 52 | [20, 100] | 更早切科学计数法，中文单位使用减少，沉浸感降低 | 更晚切科学计数法，但单位表不够时需扩展 |
| `SIG_DIGITS` | 3 | [2, 5] | 显示更精确，UI 占位更长 | 显示更粗略，信息损失 |
| `THOUSAND_SEPARATOR` | `","` | [",", ".", " ", ""] | 改变直接数字的分隔符样式 | — |
| `ZERO_DISPLAY` | `"0"` | 任意非空字符串 | 改变零值显示文本 | — |
| `MAX_DISPLAY` | `"MAX"` | 任意非空字符串 | 改变上限值显示文本 | — |
| `TRIM_TRAILING_ZEROS` | `false` | [true, false] | `"1.5万"` 而非 `"1.50万"`，更简洁但列宽不一致 | 保持固定列宽，数字对齐更好 |

说明：这些参数主要影响 UI 显示，不影响任何数值计算逻辑。可在设置系统中暴露给玩家（MVP 后期），但不影响游戏平衡。
